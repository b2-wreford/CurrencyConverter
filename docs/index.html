Advantages/disadvantages of the REST compaired to SOAP and XML RPC?

Firstly, REST is easier to understand and use for developers as it uses HTTP (and other protocols such as FTP, SMTP) with operations that are already familiar to us such as POST, PUT, GET and DELETE for transferring data. It is much easier for developers and web users take advantage of API’s that are RESTful, which is why all major web companies now use REST and encourage its use.

Another great advantage of using REST is that it is stateless which means that only the data is transferred between server and client which means you can separate the two concerns; the server is no longer worried about the user interface and the client should not be worried about data storage which then allows for greater scalability, independent deployment of components and allows legacy systems to be encapsulated and still work using a RESTful system. REST is lightweight in nature and is much more suited to a mobile web where connection and bandwidth may vary. REST also allows for caching, again increases performance and using less bandwidth which SOAP cannot do.

SOAP and REST vary also in the way that they function with REST being more of an architecture for accessing a resource through a URI whereas SOAP is a protocol for transferring data that’s more function-driven, whereas REST is data-driven.

REST also permits many different data formats such as plain text, HTML, XML, and JSON whereas SOAP only allows XML.

Firstly client side validation on my form would be one method for providing a better, more intuitive solution for catching user errors. This could be done through javascript validation on the input fields that could check my inputs and provide error messages even before the form has been submitted. This paired would server side validation would be a rigorous and professional solution for my application.

Although it features it already, there is room for more functional programming techniques to be applied in my code. This is a programming paradigm that is a way of thinking about software construction that allows for more concise, more predictable and easier to test code. This could be applied to both JavaScript and PHP coding within my project.

Another way in which the project could be extended would be to use a framework such as Node.js deployed on the server to handle all of file requests. Node.js is built on javascripts asynchronous processing that can service more concurrent requests than conventional multithreaded servers. This makes it a great fit for data driven applications – which is slightly beyond the scope of our project but is worth noting for future developments. It would also then mean all of the programming code would be written in one language.
